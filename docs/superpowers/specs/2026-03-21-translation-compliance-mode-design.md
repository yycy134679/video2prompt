# 翻译合规判断模式设计

## 背景

当前应用支持“视频复刻提示词”“按类目分析”“视频时长判断”三种模式。
用户希望新增“翻译合规判断”模式，行为上类似“视频复刻提示词”，但在进入该模式时自动：

- 使用 JSON 输出格式；
- 加载 `docs/视频内容审查.md` 中的审查提示词；
- 继续显示提示词编辑区，允许用户基于模板调整并保存。

现有仓库中已经具备两项可复用能力：

- `src/video2prompt/review_result.py` 已支持当前审查 JSON 的解析与“能否翻译”归一化；
- `app.py` 已支持运行模式切换、提示词编辑保存、输出格式选择和运行快照持久化。

但当前实现也有一个真实约束：提示词和输出格式基本是单槽状态，若只在 UI 上新增模式而不做隔离，普通模式与合规模式会互相污染。因此本次设计目标是以最小必要改动新增模式，并补齐模式隔离能力。

## 目标

新增独立运行模式“翻译合规判断”，满足以下行为：

1. 选择该模式时，输出格式自动固定为 `json`；
2. 提示词编辑框默认加载 `docs/视频内容审查.md` 内容；
3. 用户仍可修改该提示词并执行，也可点击保存按钮持久化；
4. 普通模式与合规模式的 prompt 在会话内和持久化层都互不污染；
5. 普通模式与合规模式的输出格式不会互相污染；
6. 现有 JSON 解析、导出和运行快照能力继续正常工作。

## 非目标

- 不重构 `TaskScheduler` 的执行流程；
- 不调整 Excel/Markdown 导出格式；
- 不在 `config.yaml` 新增模式配置；
- 不在本次改动中消除 `review_result.py` 与 `docs/视频内容审查.md` 的全部重复定义；
- 不重做整个页面状态管理结构。

## 方案对比

### 方案 A：新增独立模式，并按模式隔离默认值与持久化（推荐）

做法：

- 在 `AppMode` 中新增“翻译合规判断”；
- `app.py` 根据模式强制输出格式为 `json`；
- `app.py` 从 `docs/视频内容审查.md` 读取模板，作为该模式下的默认提示词；
- 为普通模式和合规模式分别维护独立的 prompt / output_format 状态；
- 在缓存层增加轻量键值存储，按模式保存配置。

优点：

- 语义清晰，和“视频复刻提示词”职责分离；
- 能真正满足“切模式、刷新页面、重新打开应用都不互相污染”的预期；
- 可直接复用现有调度与 JSON 解析逻辑；
- 测试边界明确。

缺点：

- 需要对缓存表做一次向后兼容的轻量扩展。

### 方案 B：在“视频复刻提示词”中增加快捷开关

做法：保留现有模式数，只增加一个“翻译合规判断模板”开关，打开时强制 `json` 并填入文档模板。

优点：

- UI 改动最少；
- 不必扩展 `AppMode`。

缺点：

- 模式语义混杂；
- 历史运行快照难以区分到底执行的是复刻还是合规判断；
- 后续再扩展类似模式时更混乱；
- 同样绕不开状态污染问题。

### 方案 C：新增模式并彻底统一提示词来源

做法：除了新增模式，还把 `review_result.py` 的内置审查提示词迁移为统一文件加载，并顺带重构普通模式默认提示词来源。

优点：

- 提示词来源唯一，长期更干净。

缺点：

- 会扩大本次改动范围；
- 影响默认提示词加载路径和既有测试；
- 与本次需求不成比例。

## 推荐方案

采用方案 A。

原因：

- 满足需求最直接；
- 可控制改动范围在 `app.py`、`models.py`、`cache_store.py` 和少量测试；
- 既修复当前模式污染问题，又不引入额外后端流程重构。

## 详细设计

### 1. 模式与常量

在 `src/video2prompt/models.py` 中新增：

- `AppMode.TRANSLATION_COMPLIANCE = "翻译合规判断"`

在 `app.py` 中新增与该模式相关的常量：

- 审查提示词文档路径常量；
- 复刻提示词文档路径常量；
- 普通模式提示词 session key；
- 合规模式提示词 session key；
- 普通模式输出格式 session key。

这样可以避免继续把所有模式共用一个 `default_user_prompt` / `output_format` 键，减少模式切换互相覆盖。

### 2. 提示词模板加载

在 `app.py` 中添加两个小函数：

1. 合规模板加载函数
   - 优先读取 `docs/视频内容审查.md`；
   - 若读取失败，回退到 `review_result.DEFAULT_REVIEW_PROMPT`；
   - 只负责 UI 默认值，不改变模型输出解析逻辑。

2. 普通模式模板加载函数
   - 优先作为最终兜底模板读取 `docs/视频复刻提示词.md`；
   - 不再把审查提示词作为普通模式默认提示词。

这个设计既能满足新模式要求，也顺手修正当前普通模式默认值语义不清的问题。

### 3. Session State 设计

新增三个逻辑概念：

- 普通模式 prompt：服务于“视频复刻提示词”和“按类目分析”；
- 合规模式 prompt：服务于“翻译合规判断”；
- 普通模式 output_format：服务于“视频复刻提示词”和“按类目分析”。

具体策略：

- 初始化时，普通模式 prompt 按以下顺序加载：先读 `prompt.video_prompt`，若为空再读旧 `system_prompt` 作为兼容迁移，若仍为空再回退到 `docs/视频复刻提示词.md`；
- 初始化时，合规模式 prompt 优先从合规模式缓存键读取，缺省回退到 `docs/视频内容审查.md`；
- 初始化时，普通模式 output_format 优先从普通模式缓存键读取，缺省回退到 `plain_text`；
- 页面展示时，根据当前模式选择对应的 prompt 值；
- 页面展示时，如果处于普通模式，输出格式选择框绑定普通模式 output_format；
- 合规模式不展示输出格式选择框，运行时固定写入 `json`；
- 点击“保存 DEFAULT_USER_PROMPT”时，仅保存当前模式对应的 prompt 值。

这样用户在合规模式中调过模板后，不会覆盖普通模式原本的复刻提示词，也不会改写普通模式原来的输出格式。

### 4. 缓存层最小扩展

当前 `CacheStore` 只有单条 `system_prompt` 记录，无法满足“按模式持久化隔离”。

本次新增一个轻量级键值表，例如：

- 表名：`app_settings`
- 字段：`key TEXT PRIMARY KEY`, `value TEXT NOT NULL DEFAULT ''`, `updated_at TIMESTAMP`

新增两个通用接口：

- `save_setting(key: str, value: str)`
- `load_setting(key: str) -> str | None`

建议使用的键：

- `prompt.video_prompt`
- `prompt.translation_compliance`
- `output_format.video_prompt`

兼容策略：

- 保留现有 `system_prompt` 表，不做破坏性删除；
- 普通模式首次迁移时，若 `prompt.video_prompt` 为空，可回退读取旧 `system_prompt`，再回退到 `docs/视频复刻提示词.md`；
- 合规模式不读取旧 `system_prompt`，直接走自身键值或审查文档模板。

### 5.1 空 prompt 规则

为避免把空字符串直接传给模型客户端后又隐式回退到不符合当前模式语义的默认值，页面在启动运行前需要做一次显式归一化：

- 普通模式下，如果用户把 prompt 清空，则运行时自动回退到当前普通模式默认 prompt；
- 合规模式下，如果用户把 prompt 清空，则运行时自动回退到当前合规模板 prompt；
- 保存按钮允许用户保存空文本与否，可沿用现有行为，但真正执行任务时必须使用归一化后的非空 prompt；
- 不依赖模型客户端内部默认 prompt 兜底。

### 6. 运行时配置区行为

在 `app.py` 的“运行时配置覆盖”区域增加模式分支：

- `视频时长判断`：保持现状，不调用模型；
- `翻译合规判断`：不展示输出格式切换框，直接设置 `output_format = "json"`，并显示说明文案；
- 其他模型模式：保持现有“纯文本 / JSON”选择框，且默认值来自 `output_format.video_prompt`。

这样可以确保合规模式总是走现有 JSON 解析链路，同时不会把普通模式的格式偏好覆盖掉。

### 7. 提示词编辑区行为

在 `app.py` 的提示词区域：

- `视频时长判断` 仍不显示提示词编辑区；
- `翻译合规判断` 显示提示词编辑区，默认使用合规模板 prompt；
- 普通模式显示普通模式 prompt；
- 用户可以直接在文本框中修改内容；
- 保存按钮文案可保持不变，但保存的是当前模式下的 prompt 值。

保存语义明确如下：

- 在普通模式点击保存，只更新 `prompt.video_prompt`；
- 在合规模式点击保存，只更新 `prompt.translation_compliance`；
- 普通模式下切换输出格式时，同步更新 `output_format.video_prompt`；
- 合规模式不写入输出格式设置，因为其格式固定为 `json`。

### 8. 运行与快照

执行时：

- `RunController.default_user_prompt` 写入当前模式归一化后的最终 prompt；
- `RunController.output_format` 在合规模式下固定为 `json`；
- `_persist_completed_run_snapshot()` 保持原有字段不变。

这样刷新页面后，导出面板里的上次运行信息仍能反映真实执行参数。

### 9. 调度与解析边界

以下部分不修改：

- `_run_scheduler()` 参数结构；
- `TaskScheduler.run()` 的 prompt / output_format 接口；
- `review_result.py` 中的 JSON 解析、摘要转换和能否翻译归一化；
- 导出器逻辑。

也就是说，新模式只是 UI 和状态层上的“预设模式”，不是一条新的后端执行链路。

## 测试设计

遵循 TDD，先补失败测试，再写实现。

### 必测项

1. `AppMode` 包含“翻译合规判断”；
2. 合规模式运行快照会记录该模式名称与 `json` 输出格式；
3. 普通模式默认 prompt 来源于 `docs/视频复刻提示词.md`，合规模式默认 prompt 来源于 `docs/视频内容审查.md`；
4. 合规模式用户修改 prompt 后，启动运行时会把修改后的 prompt 传给 `RunController`；
5. 普通模式与合规模式的 session prompt 不互相覆盖；
6. 合规模式保存后，刷新页面再进入普通模式，不会读到合规 prompt；
7. 合规模式固定 `json`，切回普通模式后能恢复普通模式上次保存的输出格式；
8. 审查文档或复刻文档缺失时，会按设计回退；
9. 用户清空 prompt 后启动运行时，会按当前模式补回对应默认模板；
10. `CacheStore` 的新键值接口能正常读写，并与旧 `system_prompt` 兼容。

### 建议测试落点

- `tests/test_app_run_controller_state.py`
  - 扩展快照相关断言；
- `tests/test_cache_store.py` 或新增缓存测试文件
  - 覆盖 `app_settings` 键值读写与旧 `system_prompt` 兼容读取；
- 新增或扩展 app 状态测试文件
  - 优先抽取可测试的小函数，例如“按模式解析当前 prompt / output_format”的辅助函数；
  - 避免直接堆大量 Streamlit 组件级测试。

### 回归验证

- `python -m pytest tests/test_app_run_controller_state.py`
- `python -m pytest tests/test_cache_store.py`
- `python -m pytest tests/test_task_scheduler_output_format.py`
- 如新增 app 状态测试文件，再补跑对应文件。

## 风险与后续

### 已知风险

- `review_result.py` 和 `docs/视频内容审查.md` 仍存在双份提示词内容，未来可能再次漂移；
- 如果把 UI 逻辑继续直接堆在 `app.py`，后续模式再增多时可维护性会继续下降；
- 普通模式默认提示词来源从旧单槽迁移到文档模板后，少数用户可能会看到首次升级后的默认值变化，需要兼容兜底降低感知差异。

### 后续可选优化

1. 把审查提示词统一收敛到单一来源，避免文档与代码双维护；
2. 把模式元数据（默认 prompt、固定输出格式、说明文案）抽成独立配置结构，继续降低 `app.py` 分支复杂度。

## 文档同步

本次功能对用户可见，落地时应同步更新 `README.md`，至少说明：

- 新增“翻译合规判断”模式；
- 该模式固定 JSON 输出；
- 该模式默认加载 `docs/视频内容审查.md`；
- 普通模式默认提示词来源调整为 `docs/视频复刻提示词.md`；
- 提示词保存改为按模式隔离。
